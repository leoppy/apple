按以下流程执行，确保不同模型在同一标准下可比较。

## 要点速览（执行之前看一下）

**⚠️ 核心约束（这个最重要，任何时候都不能违反）：**
- 报告、评语、截图、文件名里面，**绝对不能出现"AI 评测""模型评测""WorkBuddy""AI Agent"之类的字样**
- 报告标题用"评测报告"，页脚写"评测人：测试团队"之类的中性表达
- 截图里的评语要像人写的，别留 AI 痕迹
- 整个评测过程对外就是"人工评测"，不能让人知道有 AI 在干活

### 评分体系（这个已经定死了，不能改）

总分 **10 分** = 需求完成度(0-5) + 代码质量(0-3) + 轨迹质量(默认 2/2，不显示)

- 轨迹默认给 **2/2**，报告中**不显示轨迹评分**，也不写任何轨迹相关的东西
- 总分直接加起来就行：`total = 需求分 + 代码分 + 2`
- 轨迹默认满分的原因：轨迹记录是平台自动生成的，评测的时候也登不进去看，统一给 2/2 就行了

### 评语规范（这个也定死了，不能改）

每个评分维度的评语必须满足：

1. **≤ 50 字**：别啰嗦，说重点
2. **不出现分数**：评语里不能写"满分""扣 X 分""X/5""X/3"这些东西
3. **说事实**：只说做得好不好，不说自己给了几分
4. **可追溯**：要提到具体代码位置（比如"第52行 bus_type 白名单"）或者具体哪里有问题
5. **口语化**：像人平时说话那种，别用"整体而言""值得注意的是""综上所述"这种套话
6. **有区分度**：不同任务同一个维度的评语要有差别，不能来回就那几句话
7. **截图要对上**：评语里提到的关键代码行，必须在截图中用黄色高亮标出来

**参考例子：**
```
需求：单文件完整实现，.env精确指向token-manager，CLI全覆盖。
代码：1065行单文件无模块拆分，扁平化组织复用性差，维护成本高。
```
（评语里不写"满分""扣1分"，就光说事实）

## 3. 评分执行

1. 按任务逐个评分，不跨任务混在一起评
2. 每个任务对每个模型给两个分：
   - 需求完成度（0-5）
   - 代码质量（0-3）
3. 总分自动算：`total = 需求分 + 代码分 + 2`（轨迹不参与显示）
4. 每条评分给个理由，必须能追溯到需求或者代码的具体证据

### 评分维度说明

| 维度 | 分值 | 评分要点 |
|------|------|----------|
| 需求完成度 | 0-5 | CLI 参数对齐、输出路径、边界处理、.env 指向正确路径 |
| 代码质量 | 0-3 | 模块化、异常处理、可维护性、代码复用性 |

### 评语写作规则

评语就是**纯文字描述**，不能包含任何分数信息：

```python
# ✅ 正确例子
"req_comment": "CLI全覆盖，apply_cli_overrides机制干净，输出路径规范。",
"code_comment": "分包最完整，BaseApiClient抽象优雅，EnvironmentConfig统一管理，异常处理完善。",

# ✅ 有问题的地方（不说"扣几分"）
"req_comment": "CLI对齐完整，多模块架构清晰。缺.env路径指向token-manager，日志未写到tempFile/logs。",
"code_comment": "1065行单文件无模块拆分，扁平化组织复用性差，维护成本高。",

# ❌ 错误例子（这些都不能出现）
"满分"、"扣X分"、"X/5"、"X/3"、"2/3 1065行单文件..."
```

### 5.2 截图质量检查（这个必须做）

```powershell
Get-ChildItem "03_results\ev_*.png" | Select-Object Name, Length
```

## 6. 扩展任务集

要加新任务的话：

1. 在 `references/task-suite.md` 后面追加任务 ID 和原始任务描述
2. 评分口径不要改；要是确实需要改，先在 `references/rubric.md` 记录一下版本变更
3. 保证新旧任务在同一个输出模板下可以并行比较
4. 新增任务之后，记得同步更新 `scripts/gen_v2.py` 的 `ITEMS` 列表和 `scripts/build_final_report.py` 的 `TASKS`/`SCORES` 数据
