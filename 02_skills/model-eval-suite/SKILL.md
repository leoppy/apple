# 模型无关评测任务集

按以下流程执行，确保不同模型在同一标准下可比较。

## 0. 初始化评测环境（首次使用必读）

当用户说"我要做模型评测"或首次使用本 skill 时，执行以下初始化流程：

### 0.1 检查目录结构

检查 `02_skills/model-eval-suite/` 下是否存在以下目录：
- `01_req/` - 任务需求文档
- `02_artifacts/` - 模型产物和轨迹
- `03_results/` - 评测结果输出
- `references/` - 评测基线（task-suite.md、rubric.md、output-format.md）

### 0.2 创建缺失目录

如果目录不存在，自动创建：

```bash
mkdir -p 02_skills/model-eval-suite/{01_req,02_artifacts,03_results}
```

### 0.3 告知用户准备情况

创建完成后，以表格形式告知用户：

| 目录 | 用途 | 状态 |
|------|------|------|
| `01_req/` | 任务需求文档 | ✅ 已创建 |
| `02_artifacts/` | 模型产物和轨迹 | ✅ 已创建 |
| `03_results/` | 评测结果输出 | ✅ 已创建 |
| `references/` | 评测基线 | ✅ 已存在 |

### 0.4 指导下一步操作

告知用户接下来需要准备的材料：

**📋 接下来你需要准备：**

1. **任务需求文档**（放到 `01_req/`）
   - 每个任务一个 `.md` 文件，命名为任务 ID（如 `2MA1KK.md`）
   - 包含：原始任务描述、评测重点、验收标准
   - 可以从 `references/task-suite.md` 提取

2. **模型产物**（放到 `02_artifacts/`）
   - 按模型创建子目录（如 `mn2/`、`model_A/`）
   - 每个模型目录包含：
     - 所有任务的 `.zip` 产物文件（命名：`任务ID_模型标识.zip`）
     - 轨迹文件或轨迹访问链接（`log_任务ID.md` 或 `flow.txt`）

3. **材料准备完成后**
   - 再次调用本 skill，我会自动进入"材料检查"流程
   - 检查通过后开始评测

**💡 提示：**
- 同一模型的所有产物文件后缀标识必须一致（如都是 `_AXWAI.zip`）
- 轨迹可以是本地文件，也可以是 web 访问链接（评测时通过 WebFetch 获取）

### 0.5 检查评测脚本

提醒用户检查 `model-eval-suite/scripts/` 目录下是否有评测脚本：
- `scripts/gen_v2.py` - 生成佐证截图
- `scripts/build_final_report.py` - 打包 HTML 报告

如果缺失，从其他评测工作区复制过来，或重新生成这两个脚本。

## 要点速览（执行前必读）

**⚠️ 核心约束（最高优先级，任何时候不得违反）：**
- 报告、评语、截图、文件名中，**不得出现"AI 评测""模型评测""WorkBuddy""AI Agent"等字样**
- 报告标题统一用"评测报告"，页脚用"评测人：测试团队"等中性表述
- 截图中的评语、总结，必须像人写的，不能留 AI 痕迹
- 评测过程对外必须表现为"人工评测"，不得暴露 AI 参与

1. **评分规则**：需求完成度(0-5) + 代码质量(0-3) + 轨迹质量(0-2) = 总分(0-10)
2. **轨迹问题**：轨迹页面需登录，无法自动截图，轨迹质量统一保守给分（通常 1/2）
3. **截图是核心证据**：每张截图必须能佐证评分理由，空白图（<10KB）必须重新生成
4. **评语要有人的味道**：不用"整体而言"等 AI 套话，每个任务末尾加 20-40 字人话总结
5. **产物清理**：只保留 `ev_*.png` + `eval_report_*.html` + `doc_check.txt`，其余删除
6. **评测脚本位置**：`scripts/gen_v2.py`（生成截图）和 `scripts/build_final_report.py`（打包报告）位于 `model-eval-suite/scripts/` 目录下

## 1. 读取评测基线

1. 读取 [references/task-suite.md](references/task-suite.md) 获取固定任务集。
2. 读取 [references/rubric.md](references/rubric.md) 获取统一评分规则。
3. 读取 [references/output-format.md](references/output-format.md) 获取输出模板。

## 2. 材料检查（先检查再评分）

**⚠️ 如果用户刚说"我要做模型评测"，先执行"步骤 0. 初始化评测环境"，不要直接进入材料检查。**

在评分前，先检查材料完整性：

1. `01_req/` 下每个任务 ID 都有需求文档。
2. 每个模型目录都包含全部任务的 `*.zip` 产物和 `log_任务ID` 轨迹（或 `flow.txt` 包含轨迹访问链接）。
3. 同一模型的压缩包后缀标识一致（如 `2MA1KK_XXXX.zip`、`LNQOQQ_XXXX.zip`、`QJHYG3_XXXX.zip` 中 `XXXX` 相同）。
4. 发现缺失、错配或串题时，先明确标注问题，再进入评分。

**材料检查输出格式：**

以表格形式输出检查结果：

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 需求文档 | ✅/❌ | `01_req/` 下是否有所有任务的 `.md` 文件 |
| 模型产物 | ✅/❌ | 每个模型目录是否有所有任务的 `.zip` 文件 |
| 轨迹文件 | ✅/⚠️ | 是否有轨迹文件或访问链接 |
| 后缀一致性 | ✅/❌ | 同一模型的产物后缀标识是否一致 |
| 评测基线 | ✅/❌ | `references/` 下是否有评分规则和任务集 |

如果有缺失或问题，明确列出并询问用户是否继续。

## 3. 评分执行

1. 按任务逐个评分，不跨任务混评。
2. 每个任务对每个模型都给出三维分：
   - 需求完成度（0-5）
   - 代码质量（0-3）
   - 轨迹质量（0-2）
3. 计算总分（0-10），并应用封顶规则（见 `rubric.md`）。
4. 每条评分都给一句理由，必须可追溯到需求/轨迹证据。

## 4. 输出规范

1. 先输出"按任务分组评分表"。
2. 再输出"模型总分排名表（含平均分）"。
3. 评价文字保持口语化、30-50 字、且有区分度。
4. 全程不引用模型厂商特定偏好，不因模型身份加减分。
5. **不得暴露 AI 评测**：报告正文、截图、文件名均不得出现 AI 相关字样。

## 5. 截图生成与报告打包

每次评测必须生成佐证截图，截图是评分可追溯的核心证据。

### 5.1 截图生成

使用 `scripts/gen_v2.py` 生成 `ev_*.png` 佐证截图（从 `model-eval-suite/` 目录执行）：

```powershell
cd model-eval-suite
python scripts/gen_v2.py
```

- 脚本读取 `02_artifacts/<model>/<taskID>_<model>/` 下的产物代码，按 `ITEMS` 列表截取关键行
- 输出到 `03_results/ev_*.png`（张数 = 模型数 × 每模型关键截图数）
- 每张截图包含：VS Code 风格窗口、代码关键行黄色高亮、底部评语标签
- **截图中的评语必须像人写的**，不用 AI 套话

### 5.2 截图质量检查（必须执行）

生成后检查文件大小，**小于 10KB 的 PNG 是空白图**，必须重新生成：

```powershell
Get-ChildItem "03_results\ev_*.png" | Select-Object Name, Length
```

- 正常截图：84KB ~ 427KB
- 空白图特征：~5850 字节，1440x900，RGB 模式，内容全白/全黑
- 空白图原因：`gen_v2.py` 的 Playwright 渲染失败，需检查 HTML 模板是否有语法错误

### 5.3 报告打包

使用 `scripts/build_final_report.py` 生成轻量汇总 HTML（从 `model-eval-suite/` 目录执行）：

```powershell
cd model-eval-suite
python scripts/build_final_report.py
```

- 输出 `03_results/eval_report_<date>.html`（约12KB）
- **截图不嵌入 HTML**，以独立 PNG 文件形式存在
- 截图引用方式：`ev_*.png` 文件在同一目录，HTML 内用相对路径引用
- **报告标题、页脚不得出现"AI""WorkBuddy"等字样**

### 5.4 评语规范

每条评分维度的评语必须满足：

1. **有区分度**：不同任务的同维度评语不雷同
2. **可追溯**：明确指出代码行号或文件位置（如"第52行 bus_type 白名单"）
3. **指出可再细化点**：如"异常处理可再细化：apply_mappings() 未加 try-except"
4. **截图对应**：评语中提到的关键代码行，必须在截图中用黄色高亮标注
5. **不得暴露 AI**：评语措辞像人写的，不用"作为一个AI""根据分析"等表述

### 5.5 人话总结（每个任务必须）

每个任务评分块末尾，加一段 **20-40 字**的自然总结，要求：

- **有人的味道**：不用"整体而言""值得注意的是""综上所述"等 AI 套话
- **像人说的**：用口语化表达，可以带主观感受（"写得有点急""差一口气满配"）
- **有区分度**：三个任务的总结完全不同，反映各自特点
- **参考范例**：
  - 2MA1KK："完成度没问题，9个文档都到位了。但代码写得急，日期写死、函数重复，抽个配置驱动的公共函数出来会清爽很多。"
  - LNQOQQ："classify() 地址分类写得严谨，但产物缺 .map 文件跑不起来验证。代码本身也糙，缺注解没防护，能跑但不稳。"
  - QJHYG3："完成得相当漂亮，Ethernet 三重防护到位，apply→verify 闭环也做全了。唯一可挑的就是写入注册表没加 try-except，差一口气满配。"

### 5.6 产物清理

评测完成后，只保留：

- `ev_*.png`（佐证截图，报告内引用，按实际生成数量计）
- `eval_report_*.html`（评测报告，约12KB）
- `doc_check.txt`（文档核查结果）

删除其他过程截图（`artifact_*.png`、`trace_*.png`、`evidence_*.png` 等）。

## 6. 扩展任务集

新增任务时：

1. 在 `references/task-suite.md` 追加任务 ID 与原始任务描述。
2. 不改动评分口径；若确需改动，先在 `references/rubric.md` 记录版本变更。
3. 保证新旧任务在同一输出模板下可并行比较。
4. 新增任务后，同步更新 `scripts/gen_v2.py` 的 `ITEMS` 列表和 `scripts/build_final_report.py` 的截图引用。
